MailHive — Déduplication des pièces jointes

Stockage content-addressed : N mails partageant une PJ → 1 seul blob. Backend PostgreSQL ou S3/SeaweedFS via le port BlobStore.

Écriture — déduplication

  • • storeAttachments appelé 1× par PJ pour toute la campagne, pas par mail.
  • • SHA-256(contenu) → UpsertMeta sur (tenant_id, sha256).
  • • Le blob n'est écrit (Put) que si le contenu est nouveau ; sinon no-op.
  • • Mail + destinataires + liens écrits dans une seule transaction pgx (atomique).

Stockage — 3 tables + port

  • • attachments : 1 ligne / contenu (métadonnée, pas de compteur).
  • • mail_attachments : liaison N→1, porte le filename (propre au mail).
  • • attachment_blobs / S3 : les octets, clé (tenant_id, sha256), sans FK (symétrie des backends).
  • • Dédup scopée tenant : isolation, pas de fuite inter-tenant.

Envoi & GC

  • • Worker : GetAttachmentRefs → Load → reconstitue mail.Attachments avant l'envoi (le Sender est inchangé).
  • • Pas de transaction distribuée DB↔S3 : blob d'abord à l'écriture, blob d'abord à la suppression.
  • • GC @hourly : NOT EXISTS de lien → supprime blob puis métadonnée, avec délai de grâce 1h. Pas de compteur dénormalisé (donc rien qui dérive à l'archivage).
